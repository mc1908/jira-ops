# jira-ops feedback: multi-line `--description` via shell is unusable

**Skill version tested:** current (`mc1908/jira-ops`)
**Platform:** Windows / PowerShell 5.1
**Command affected:** `jira update` (and `jira create`) — `--description` flag
**Reported by:** min.chen — 2026-07-08

---

## Problem

`--description` accepts only a plain command-line string argument. When the
description contains multiple lines (e.g. a Jira wiki-markup story with `h2.`
headings, bullet lists, and `{{monospace}}` spans), passing it through any
shell becomes impractical or impossible without a workaround:

- **PowerShell here-strings** (`@"…"@`) combined with backtick line-continuation
  are silently mangled by the agent's `run_in_terminal` tool, producing truncated
  or garbled output with no error message.
- **Inline newlines / escaping** (`\n`, `` `n ``, `$([char]10)`) require
  double-escaping that is fragile and hard to review.
- **Quoting multiline strings in a single shell line** is not reliably portable
  across cmd, PowerShell, and bash.

The practical result: any write operation that needs a non-trivial description
(a user story, acceptance criteria, retrospective note) requires the user or the
agent to first author a **temporary Python wrapper script** that bypasses the
shell entirely — which is exactly what happened during a live session.

---

## What the agent had to do as a workaround

Instead of running:

```powershell
python scripts/jira.py update ALGONQUIN-1234 `
  --summary "..." `
  --description "...multi-line wiki markup..." `
  --dry-run
```

…the agent created a temporary file at `%TEMP%\jira_poc_update.py` that
hardcoded the description and called `jira.py` via `subprocess`, then had to be
run separately with a `--apply` flag to switch between dry-run and live mode:

```python
# C:\Users\...\AppData\Local\Temp\jira_poc_update.py  (generated by agent)
import subprocess, sys, pathlib

DESCRIPTION = """\
h2. Goal
...six scenarios of Given/When/Then...
h2. Estimate
Phase A only: 5-8 person-days
"""

cmd = [str(PYTHON), str(JIRA_PY), "update", "FASTACTR67-578",
       "--summary", SUMMARY, "--description", DESCRIPTION]
if "--apply" not in sys.argv:
    cmd.append("--dry-run")
subprocess.run(cmd, ...)
```

This is a three-step detour (write file → preview → apply) for what should be a
single `--dry-run` / confirm / run workflow.

## Suggested improvements (pick one or combine)

### Option A — `--description-file <path>` flag  *(preferred)*

Read the description from a file path instead of a CLI string. The agent can
write the content to a temp file safely (no shell quoting involved) and then
pass the path:

```bash
# Agent writes description to file, then:
python scripts/jira.py update ALGONQUIN-1234 \
  --summary "..." \
  --description-file "$env:TEMP\desc.txt" \
  --dry-run
```

This is the cleanest approach: the file is the source of truth, the shell only
sees a path string, and the content round-trips losslessly.

### Option B — `--description -` reads from stdin

Allow `-` as a sentinel to read the description from stdin:

```bash
cat desc.txt | python scripts/jira.py update ALGONQUIN-1234 \
  --summary "..." --description - --dry-run
```

Familiar Unix convention; works on all platforms. Requires the agent to write
the file first (or use a PowerShell pipe), but avoids any new flag name.

### Option C — positional or `--field-file` escape hatch

Extend the existing `--field-json` escape hatch to also accept a file path:

```bash
python scripts/jira.py update ALGONQUIN-1234 \
  --field-json-file "$env:TEMP\payload.json" --dry-run
```

Less ergonomic but covers all fields at once (useful for `create` too).

---

## Acceptance criteria for the fix

```
Given a description containing multiple lines, Jira wiki-markup headings,
  bullet lists, and {{monospace}} spans
When the user (or an agent) runs `jira update KEY --description-file path`
  (or equivalent) with --dry-run
Then the dry-run output shows the full multi-line description without
  truncation, escaping artefacts, or errors
  AND the live write stores the description in Jira identically to the file
  content
```

---

## Notes

- The issue is **not** specific to Windows; any platform where the shell
  transforms quotes/newlines before the Python process sees argv will hit this.
- The dry-run preview (`[dry-run] Would update …`) already works correctly once
  the description reaches the Python layer — so the fix is entirely in argument
  ingestion, not in the REST payload or formatting code.
- A `--description-file` flag would also benefit the `create` command, which
  shares the same `--description` argument pattern.
